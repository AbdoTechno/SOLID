# أسئلة إنترفيو عن مبدأ فصل الواجهات: interview-questions.md 💼

مبدأ الـ ISP هو سر رشاقة الكود. دي أكتر من 20 سؤال متدرجة الصعوبة مع إجابات نموذجية بالعامية المصرية:

---

## 👶 مستوى جونيور (Junior Level) - أسئلة أساسية

### س1: يعني إيه ISP بالبلدي؟
**الجواب**: يعني ما تجبرش كلاس إنه يعمل Implement لواجهات أو ميثودس هو مش محتاجها. الصح إننا نقسم الواجهات الكبيرة لواجهات صغيرة متخصصة.

### س2: إيه المشكلة لو عملت Interface واحدة فيها كل حاجة؟
**الجواب**: المشكلة إن الكلاسات اللي هتطبقها هتضطر تكتب ميثودس فاضية أو ترمي Exceptions لمجرد إرضاء الـ Compiler، وده بيخلي الكود صعب القراءة ومزعج في الصيانة.

### س3: هل الـ ISP بيركز على الكلاسات ولا الـ Interfaces؟
**الجواب**: الـ ISP بيركز بشكل أساسي على الـ Interfaces (الواجهات) وطريقة تصميمها عشان تكون رشيقة ومريحة للكلاسات اللي بتستخدمها (Clients).

### س4: لو عندي كلاس بيرمي `UnimplementedError` في ميثود معينة، ده معناه إيه؟
**الجواب**: ده أوضح مؤشر (Red Flag) على إن السيستم كاسر مبدأ الـ ISP ومبدأ الـ LSP برضه. الكلاس ده اتفرض عليه يورث أو يطبق ميثود مش بتاعته.

### س5: إزاي بنحل مشكلة الـ Fat Interface؟
**الجواب**: بنحلها بإننا نكسر الـ Interface دي لكذا Interface صغيرة (مثلاً بدل `Worker` نخليها `Workable` و `Feedable`).

### س6: هل لغة Dart بتدعم الـ Interface ككلمة مفتاحية مستقلة؟
**الجواب**: لا، في لغة Dart مفيش كلمة `interface` مستقلة زي جافا. أي كلاس عادي أو Abstract Class نقدر نستخدمه كـ Interface باستخدام كلمة `implements`.

### س7: يعني إيه "واجهة بدينة" (Fat Interface)؟
**الجواب**: هي واجهة شايلة ميثودس كتيرة جداً وملهاش علاقة ببعض، وبتخدم كذا نوع من الـ Clients في نفس الوقت.

## 👨‍💻 مستوى ميد-ليفل (Mid-Level) - أسئلة متوسطة وعملية

### س8: إيه الفرق بين SRP و ISP؟
**الجواب**: 
- الـ **SRP** بيركز على الكلاسات والـ Modules (كلاس واحد = مسؤولية واحدة).
- الـ **ISP** بيركز على الـ Interfaces والـ Contracts (ما تجبرش العميل على ميثودس مش محتاجها).
المبدأين بيخدموا نفس الفكرة وهي تقليل الـ Coupling وزيادة الـ Cohesion بس على مستويات مختلفة.

### س9: إزاي بنطبق الـ Interface Composition (دمج الواجهات) في Dart؟
**الجواب**: بنطبقها بإن الكلاس يعمل Implement لكذا واجهة مع بعض يفصل بينهم فاصلة (مثال: `class TA implements StudentRole, TeacherRole`). أو باستخدام الـ `with` مع الـ Mixins لتجميع السلوكيات.

### س10: هل تقسيم الواجهات لقطع صغيرة جداً مش بيزود تعقيد السيستم؟
**الجواب**: لا، طالما التقسيم منطقي وبيخدم البيزنس. الواجهات الصغيرة بتخلي الكود متفكك ومرن (Decoupled)، وتقدر تعمل تيست لكل قطعة لوحدها وتدمجهم بسهولة. التعقيد بييجي من الواجهات الكبيرة المترابطة اللي بتخرب الصيانة.

### س11: لو عدلت ميثود في Interface عملاقة، إيه تأثير ده على كلاسات السيستم؟
**الجواب**: كل الكلاسات اللي عاملة implement للواجهة دي هتضطر تفتح ملفاتها وتعدل كودها عشان يتوافق مع التعديل الجديد، حتى الكلاسات اللي مش بتستخدم الميثود دي أصلاً! ده بيزود الـ compile time ومخاطر الـ bugs.

### 12: إزاي الـ ISP بيساعد في الـ Mocking أثناء الـ Testing؟
**الجواب**: لما تكون الـ Interface صغيرة ومحددة، كتابة الـ Mock class للتيست بتبقى سهلة جداً وسريعة. مش هتضطر تعمل Mock لـ 50 ميثود أنت مش محتاجهم في التيست بتاعك، هتكتب سطرين وبس.

### س13: لو شغال على كود قديم ولقيت Interface ضخمة بس مستخدمة في 100 مكان، هتصلحها إزاي تحت مبدأ الـ ISP؟
**الجواب**: مش هكسر الـ Interface القديمة فوراً عشان الكود ما يضربش. هعمل الـ Interfaces الصغيرة الجديدة الأول. وبعدين هخلي الـ Interface القديمة تورث (extends) من الواجهات الصغيرة دي (كدة حافظنا على الـ Backward Compatibility). وبعدين تدريجياً هغير الكود الجديد عشان يعتمد مباشرة على الواجهات الصغيرة.

### س14: إيه هي علاقة الـ ISP بمبدأ الـ Client-Specific Interface؟
**الجواب**: هما نفس الشيء. الـ ISP بيقول إن الواجهة لازم تتصمم بناءً على احتياجات العميل اللي هيستخدمها (Client-Specific) مش بناءً على الكلاس اللي بينفذها.

## 👑 مستوى سنيور (Senior Level) - أسئلة تصميمية ومعمارية

### س15: إزاي الـ ISP بيأثر على الـ Binary Compatibility والـ Shared Libraries؟
**الجواب**: في اللغات اللي بتعتمد على الـ Static linking والـ Compilation للـ Binary (زي C++ أو Go)، الـ ISP بيمنع كسر الـ Binary Compatibility. لو الواجهات ضخمة وأي تعديل فيها بيغير الـ vtable (virtual table)، المكتبات التانية المعتمدة عليها هتضطر يتعملها Recompile. تقسيم الواجهات بيحمى الـ ABI (Application Binary Interface) من التغيرات غير الضرورية.

### س16: اشرح كيف تستخدم الـ Segregated Interfaces في تصميم الـ Clean Architecture؟
**الجواب**: في الـ Clean Architecture، بنستخدم الـ ISP عند حدود الطبقات (Layer Boundaries). مثلاً، في الـ Use Cases (Application Layer)، بنقسم الـ Output Boundary (Presenter Interface) لـ Interfaces صغيرة زي `UserPresenter` و `ErrorPresenter` بدل ما نعمل Presenter واحد ضخم. ده بيخلى طبقة الـ UI حرة تطبق بس العرض اللي محتاجاه وتتجنب معرفة تفاصيل تانية.

### س17: هل يمكن تطبيق الـ ISP في لغات البرمجة الديناميكية (Dynamic Languages) زي JavaScript أو Python؟ وليه؟
**الجواب**: نعم، مع إن اللغات دي معندهاش Compiler بيجبرك تعمل implement للميثودس (Duck Typing)، بس الـ ISP مهم فيها معمارياً. لو الواجهة الافتراضية للـ Module ضخمة، المطورين هيكتبوا كود بيعتمد على ميثودس كتيرة، وده بيزود الـ Coupling ويصعب عملية الـ Refactoring أو استبدال الموديول بموديول تاني. الـ ISP هنا بيحافظ على الـ Clean Contracts.

### س18: اشرح العلاقة المعمارية بين الـ ISP والـ ISP-driven patterns كـ Role Interfaces؟
**الجواب**: الـ Role Interface هو نمط تصميمي بينتج مباشرة عن الـ ISP. بدلاً من تصميم الواجهات بناء على الكيانات (Header Interfaces زي `class Student`), بنصممها بناء على الأدوار اللي بيلعبها الكيان في سياق معين (Role Interfaces زي `Attendee`, `Payer`, `Enrollable`). ده بيخلى السيستم مرن جداً وبيركز على الأدوار والتعاون بين الكائنات (Collaboration).

### س19: لو بتصمم Microservices API (REST or gRPC)، إزاي بتطبق مبدأ الـ ISP عليها؟
**الجواب**: بنطبقه بإننا نتجنب عمل "سيرفس واحدة عملاقة" بتخدم كل الـ Clients (Web, Mobile, Third-Party). بنطبق نمط **BFF (Backend For Frontend)** بحيث نعمل API Gateway أو Endpoint مخصصة ومفصلة لكل عميل حسب احتياجاته، وبكدة بنمنع الـ Mobile client من إنه يستقبل أو يتعامل مع داتا وميثودس ضخمة خاصة بالـ Admin Web Portal.

### س20: إزاي بيتقاطع مبدأ الـ ISP مع مبدأ الـ Interface Pollution في لغات الـ OOP؟
**الجواب**: الـ Interface Pollution بيحصل لما المطور يبالغ في تطبيق الـ ISP ويعمل Interface لكل ميثود منفردة في السيستم (Over-segregation). ده بيغرق الكود في بحر من التجريدات ملهاش لازمة وبتصعب تتبع تدفق العمليات. السنيور الشاطر بيوازن وبيجمع الميثودس المترابطة بقوة (High Cohesion) في واجهة واحدة طالما بتخدم نفس الدور والـ Client.
