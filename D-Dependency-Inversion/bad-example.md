# شرح الكود السيء (العك): bad-example.md ❌

في ملف الكود [bad.dart](code/bad.dart)، يوضح هذا المثال مشكلة التعشيق المباشر وكيف يؤثر على مرونة النظام:

---

## 🧐 إيه العك اللي حصل هنا؟

كلاس `StudentEnrollmentSystem` هو موديول عالي المستوى (High-level) لأنه بيشيل البيزنس لوجيك الحساس بتاع تسجيل الطلاب. 
وكلاس `MySqlDatabase` هو موديول منخفض المستوى (Low-level) لأنه بيتعامل مع تفاصيل مادية وتكنولوجية بحتة (الاتصال بالـ DB وحفظ البيانات).

في التصميم ده، كلاس البيزنس عمل حاجتين غلط:
1. اعتمد مباشرة على الكلاس المادي `MySqlDatabase` كـ Type:
   `late final MySqlDatabase _database;`
2. قام بإنشاء كائن الـ Database بنفسه جوة الكونسركتور بتاعه:
   `_database = MySqlDatabase();`

كدة كلاس البيزنس بقى ملحوم ومربوط بقوة (Tightly Coupled) بالـ MySQL.

## 🛑 الكوارث والمشاكل اللي هتحصل قدام (Maintenance Issues)

1. **مستحيل تغير تكنولوجيا قاعدة البيانات**:
   تخيل لو الجامعة قررت تنقل كل بياناتها لـ MongoDB (NoSQL) عشان السرعة والتحجيم. 
   عشان نعمل التغيير ده، هنضطر نعمل الآتي:
   - نفتح ملف `StudentEnrollmentSystem.dart` (ملف البيزنس).
   - نمسح نوع `MySqlDatabase` ونستبدله بـ `MongoDatabase`.
   - نعدل كود الكونسركتور والـ initialization.
   - نعدل الميثودس اللي بتنادي الحفظ.
   كدة إحنا كسرنا الـ OCP لأننا عدلنا في كود البيزنس المستقر عشان نغير تفصيلة مادية! وأي لمسة في الكلاس ده ممكن تبوظ السيستم كله.

2. **صعوبة الـ Unit Testing (شبه مستحيل)**:
   لما تيجي تختبر ميثود `enrollStudent` وتتأكد إنها بتشتغل صح، الميثود غصب عنها هتفتح اتصال حقيقي بقاعدة بيانات MySQL وتحاول تحفظ! 
   - لو سيرفر الـ Database واقع، التيست هيفشل مع إن كود البيزنس بتاعك سليم!
   - التيست هيبقى بطيء جداً لأنه بيتعامل مع داتا بيز حقيقية.
   - مش هتعرف تعمل Mock لقاعدة البيانات لأن كلاس البيزنس هو اللي بيعملها `new` داخلياً ومفيش طريقة تباصي ليه Mock object من برة.
