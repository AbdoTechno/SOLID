# كبسولات الإنترفيو للمطورين المتوسطين: INTERVIEW-NOTES/mid-level.md 👨‍💻

يا هلا بيك يا صديقي في مستوى الـ Mid-Level. هنا الأسئلة بتتحول من "يعني إيه المبدأ؟" لـ "إزاي بتطبقه في مشروعك؟" و"إزاي بتعمل Refactoring لكود عك؟". دي أهم الكبسولات والأسئلة لسيناريوهات الإنترفيو دي.

## 🏗️ الربط بين مبادئ SOLID وأنماط التصميم (Design Patterns)

في الإنترفيو بيحبوا يربطوا المبادئ بالأنماط الشهيرة:
* **الـ SRP والـ Repository Pattern**: بنستخدم الـ Repository لفصل لوجيك الداتا والـ Persistence عن كلاسات الدومين والـ UI.
* **الـ OCP والـ Strategy Pattern / Factory Pattern**: بنستخدم الـ Strategy عشان نغير الخوارزمية (زي طرق الدفع أو حساب الخصومات) في الـ Runtime، وبنستخدم الـ Factory لإنشاء الكائنات المتوافقة من غير ما نغير كود الطلب الرئيسي.
* **الـ LSP والـ State Pattern**: بيضمن إن كل State object بيسد مكان الأب من غير ما يضرب السيستم بـ exceptions غريبة في الـ runtime.
* **الـ ISP والـ Role Interfaces**: بنقسم الـ Interfaces الكبيرة لـ Roles صغيرة (مثل `Enrollable`, `Payable`) وبنعمل تركيب (Composition) للأدوار دي جوة الكلاسات.
* **الـ DIP والـ Dependency Injection (DI) Frameworks**: استخدام الـ Service Locator أو الـ IoC Container لحقن الاعتماديات تلقائياً.

## 🧪 الـ SOLID وعلاقته بالـ Testing والـ Mocking

السؤال المعتاد: "إزاي تطبيق SOLID بيسهل الـ Unit Testing؟"
* **بدون SOLID**: الكود متعشق (Tight Coupling). لو حبيت تعمل Unit test لكلاس بيزنس، هيحاول يتصل بسيرفر حقيقي أو داتا بيز حقيقية، وده بيخلي التيست بطيء وهش.
* **مع SOLID (تطبيق DIP & SRP)**:
  - كلاس البيزنس بياخد الـ Interfaces بتاعته في الكونسركتور.
  - في الـ Test، بنعمل كلاس Mock بيعمل implement لنفس الـ Interface دي بس بيكتب كود وهمي خفيف (In-memory mock).
  - بنحقن الـ Mock ده جوة كلاس البيزنس، فبالتالي بنختبر لوجيك البيزنس لوحده 100% وفي أجزاء من الثانية ومن غير ما نلمس أي سيرفر حقيقي.

## 🔄 استراتيجية الـ Refactoring لكود قديم (Legacy Code Refactoring)

لو سألوك: "لو استلمت كود عك ومخالف لكل مبادئ SOLID وشغال تمام في الـ Production، إزاي بتبدأ تعمل refactor؟"
1. **لا تلمس الكود بدون Tests**: أول خطوة هي كتابة **Integration Tests** أو **Characterization Tests** لتغطية سلوك السيستم الحالي بالكامل والتأكد إنك فاهم النتائج الصح. دي شبكة الأمان بتاعتك (Safety Net).
2. **الـ Refactor التدريجي (Baby Steps)**: تجنب تماماً فكرة الـ "Big Bang Refactoring" (إنك تهد السيستم وتبنيه من الأول). كسر كلاس واحد بس في المرة.
3. **ابدأ بالـ SRP**: اسحب المسؤوليات الخارجية (الـ Database, Network, Notifications) لـ Services منفصلة.
4. **ابنِ الواجهات (Interfaces)**: ابدأ استبدل الاعتماديات المادية بالـ Abstractions تدريجياً لترسيخ الـ DIP والـ OCP.
5. **شغل الـ Tests مع كل خطوة**: اتأكد إن الـ Tests لسه بتعدي بنجاح بعد كل تعديل بسيط.

## 💬 أسئلة إنترفيو هامة للميد-ليفل

### س1: إيه الفرق بين Cohesion و Coupling؟
**الجواب**:
- **الـ Cohesion (التماسك)**: بيقيس مدى ترابط وتركيز الميثودس والمتغيرات جوة الكلاس الواحد لخدمة نفس الهدف. الصح يكون **High Cohesion** (SRP).
- **الـ Coupling (التعشيق)**: بيقيس مدى اعتماد الكلاسات على بعضها. الصح يكون **Low/Loose Coupling** عشان لو غيرنا كلاس، الكلاسات التانية ما تتبهدلش (DIP).

### س2: لو كلاس `User` فيه ميثود `toJson` و `fromJson`، هل ده كسر للـ SRP؟
**الجواب**: نظرياً نعم، لأن تحويل الداتا لـ JSON هي مسؤولية Serialization/Parsing تانية. ولكن عملياً وبشكل شائع جداً في معظم المشاريع (DTO pattern)، بنعتبرها مقبولة ومستثناة لتسهيل تبادل البيانات، طالما اللوجيك ده خفيف ومش بيعمل اتصال بالشبكة أو لوجيك معقد.

### س3: إزاي الـ Design by Contract بيساعدنا نحقق الـ LSP؟
**الجواب**: الـ Design by Contract بيفرض شروط دخول (Pre-conditions) وشروط خروج (Post-conditions) لعمليات الكلاس. الـ LSP بيشترط إن الكلاس الابن ما ينفعش يضيق شروط الدخول ولا يضعف شروط الخروج، فالتزام الأبناء بالـ Contract بيضمن أمان الإحلال بنسبة 100%.
